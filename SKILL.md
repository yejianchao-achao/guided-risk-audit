# Guided Risk Audit

## 技能定位

这是一个显式调用的重型审计技能，不是通用问答技能。

它负责三件事：

1. 半自动建立项目画像
2. 基于画像执行只读风险审计
3. 基于同一画像执行复跑与报告对比

它不负责直接修复代码。默认目标是给出结构化风险结论和最小安全修复建议，而不是立即落改动。

## 触发方式

仅在用户显式点名 `guided-risk-audit` 时触发。

不要因为用户提到“风险”“审计”“只读检查”等自然语言就自动进入本技能。

## 默认行为

- 默认以当前工作目录为审计目标
- 若用户显式指定路径，则以显式路径为准
- 默认使用简体中文输出；若用户明确要求其他语言，再切换
- 默认只读

## 运行模式

### 1. `bootstrap`

在本地还没有可用画像，或画像明显过期/缺失时使用。

目标：

- 扫描工作区并生成项目画像草稿
- 最多追问 5 轮高影响问题
- 画像达到可用状态后自动继续进入 `audit`

### 2. `audit`

在已有可用画像时使用。

目标：

- 建立变更基线
- 执行只读风险审计
- 输出结构化报告
- 缓存 `latest` 报告和时间戳归档报告

### 3. `rerun`

在用户明确要求“重跑”“复审”“二次审计”“和上次报告对比”时使用。

目标：

- 复用现有画像
- 先做画像健康检查
- 重跑只读审计
- 默认与最近一次成功报告比对
- 若用户指定历史报告，则以指定报告为对比基线

## 本地状态目录

默认状态目录：

`{CODEX_HOME}/state/guided-risk-audit/<workspace-slug>/`

其中：

- `{CODEX_HOME}` 默认为 `${CODEX_HOME:-~/.codex}`
- `<workspace-slug>` 建议由“目录名 + 绝对路径短哈希”组成，避免同名目录冲突

默认文件：

- `project-profile.md`
- `reports/latest.md`
- `reports/YYYY-MM-DD-HHMMSS.md`

## 标准流程

### 第一步：解析目标目录

1. 读取当前工作目录
2. 若用户提供显式路径，则覆盖当前工作目录
3. 若显式路径与当前上下文约束冲突，先停下确认

### 第二步：读取或建立项目画像

1. 检查状态目录下是否已有 `project-profile.md`
2. 若不存在，则进入 `bootstrap`
3. 若存在，则先做画像健康检查：
   - 路径是否仍存在
   - 仓库边界是否明显漂移
   - 默认分支候选是否变化
   - 禁改区/高风险路径是否失效
4. 若只是轻微漂移，可自动补问 1-2 个问题后继续
5. 若存在路径冲突、范围冲突、禁改区冲突等高风险歧义，必须停下确认

### 第三步：半自动建档

先自动扫描，再追问少量高影响空白项。

自动扫描优先收集：

- 单仓/多仓
- 子仓库边界
- 当前分支
- 主线基线候选
- 常见配置目录
- 生成文件痕迹
- README/启动脚本/部署脚本入口

追问规则：

- 每轮只问一个高影响问题
- 最多 5 轮
- 只问机器无法可靠判断的问题
- 优先问这些内容：
  - 本次目标切片
  - 审计范围边界
  - 配置 source of truth
  - 禁改区与高风险共享层
  - 用户可见暴露策略

5 轮后若仍有缺口：

- 把缺失项写入 `待确认`
- 进入保守审计
- 不要因为画像不完美而无限追问

### 第四步：建立基线

对每个纳入审计的仓库分别建立基线，不要在多仓聚合根做一次总 `git diff` 代替。

基线规则：

- 优先 `origin/main`
- 若不可用，再尝试 `main`
- 若项目画像提供了其他主线候选（如 `trunk` / `master`），按画像顺序尝试
- 若仍无法建立基线，标记为 `基线待确认`

必须分开检查：

1. 相对主线基线的改动
2. `staged`
3. `unstaged`
4. `untracked`

## 自动子 agent 规则

仅当当前环境和上层指令允许委派时，才自动启用只读子 agent 分片；否则退化为本地主线程分批扫描。

默认启用阈值：

- 命中 3 个及以上仓库
- 或同时覆盖前端、后端、配置面
- 或预估需要检查 200 个及以上文件

子 agent 只允许做：

- 读取代码
- 读取配置
- 读取 README/脚本
- 查看 `git status` / `git diff`
- 整理风险证据

子 agent 不允许做：

- 修改文件
- 生成 patch
- 提交 commit
- 执行会产生代码变更的命令

主进程负责：

- 分工
- 去重
- 抽样复核
- 处理冲突结论
- 最终排序与裁决

## 审计硬约束

- 默认只读，不修改文件
- 不生成 patch
- 不提交 commit
- 不把“发现问题后顺手修”当成默认动作
- 证据不足必须标记为 `待确认`
- 相对主线改动与本地未提交改动必须分开描述
- 发现生成文件问题时，必须回溯源文件与生成命令
- 发现配置问题时，必须先识别真实生效面，不能凭相似路径猜测
- 如果某项风险本来可以限制在局部私有层解决，却被落到共享默认层，默认判定为高风险共享层改动

## 风险分类

默认启用内置风险分类，见：

- `references/risk-taxonomy.md`

输出时优先归类到通用类型。只有画像明确声明项目专属风险类型，且通用类型无法准确表达时，才使用项目自定义类型。

## 输出契约

默认严格按模板输出，不走自由摘要。

固定输出顺序：

1. 审计范围摘要
2. 风险清单
3. Top 风险
4. 当前不建议动的区域
5. 最小安全修复建议

风险清单每项至少包含：

- 风险状态
- 风险级别
- 类型
- 仓库
- 变更来源
- 文件路径
- 现象
- 为什么危险
- 与本次目标的关系
- 建议的最小安全修复方式
- 证据

输出模板见：

- `templates/report.template.md`

## 项目画像契约

项目画像只写“项目事实”和“本次目标边界”，不重复写整个审计流程。

固定字段模板见：

- `templates/project-profile.template.md`

半自动追问建议顺序见：

- `templates/bootstrap-questions.md`

## 使用建议

显式调用示例：

- `使用 guided-risk-audit skill，审计当前目录的最小侵入风险`
- `使用 guided-risk-audit skill，先建项目画像，再做只读风险审计`
- `使用 guided-risk-audit skill，重跑并和上次报告比较`

## 成功标准

本技能成功的标志不是“读了很多代码”，而是：

- 建立了可复用的项目画像
- 明确区分了已证实与待确认
- 输出了结构化风险报告
- 让后续复跑可以直接复用画像与历史报告
