# Built-in Risk Taxonomy

以下为 `guided-risk-audit` 默认启用的通用风险类型。除非项目画像明确要求，否则优先使用这些类型，不要随意发明新桶。

## 1. 配置面误改

典型信号：

- 改动落在看起来相似、但并非真实生效面的配置文件
- 同一参数分散在多套配置中，但没有说明真实生效面
- 为了 demo/联调需求，把局部配置扩散到共享环境配置

## 2. 生成文件误改

典型信号：

- 直接修改 `DO NOT EDIT` 文件
- 修改生成产物，但没有回溯源文件
- README 或脚本里存在生成命令，但实际改动却落在生成结果上

## 3. 高风险共享层改动

典型信号：

- 本可在目标模块私有层解决，却改动了全站/全服务默认行为
- demo/mock/fallback 逻辑进入共享 service、共享 client、共享 UI 容器
- 局部需求推动了共享抽象漂移

## 4. mock/live 污染

典型信号：

- mock fixture、mock adapter、fallback 进入真实路径
- 真实请求失败后静默回落到 mock
- demo 常量、测试地址或临时数据进入生产 bundle 或真实接口路径

## 5. contract 漂移

典型信号：

- 前后端字段名、类型、默认值、状态机不一致
- mock contract 已变，真实接口未变，或反过来
- 页面看似可用，但提交、回填、展示会污染真实数据

## 6. 构建/环境风险

典型信号：

- 错误的 route、base URL、public path、环境变量注入
- 页面或接口只能在开发态工作
- 生产构建依赖本地开发特征或临时白名单

## 7. 用户可见信息泄漏

典型信号：

- 第三方厂商名直接暴露到前端
- provider/internal 字段进入 UI、toast、错误提示或公开接口
- 内部实现细节被包装给外部用户

## 8. 可维护性/回滚风险

典型信号：

- 临时联调写死测试地址、资源 ID、bucket、账号或环境常量
- demo 逻辑变成共享默认逻辑
- 改动很难判断影响面，未来难回滚

## 状态分类

- `已证实`：有直接代码、配置、diff、日志或运行链路证据支撑
- `待确认`：存在可疑信号，但证据不足以下结论
- `合理隔离`：确有 demo/mock/vendor 逻辑，但已被明确且安全地限制在隔离范围内
